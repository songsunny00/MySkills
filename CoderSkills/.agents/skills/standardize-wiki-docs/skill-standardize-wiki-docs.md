## 1. 需求背景

### 问题

团队的产品需求文档散落在 Wiki 中，格式混杂：

- PRD 主文档用 `.docx` 编写，含内嵌流程图和 UI 截图
- 字段规则用 `.xlsx` 维护，有多个 Sheet
- 部分文档已转换为 `.md`，但图片丢失
- 同一文档存在多个版本（如 PRD1014、PRD1108、PRD1203）

这些文档直接灌入 RAG 知识库效果很差：
- 信息不完整（图片丢失、格式错乱）
- 结构不统一（每个模块文档风格不同）
- 缺少 FAQ（用户常见问题没有标准答案）
- 不知道哪些信息缺失（产品经理没有补充清单）

### 目标

创建一个可复用的技能（Skill），让 Claude 能够：

1. **解析**混合格式的原始文档（docx/xlsx/md/图片）
2. **理解**全部内容后按标准模板**输出**结构化文档
3. **自动识别**信息缺失部分，生成**待补充清单**
4. 输出 RAG 友好的文档结构（语义分块、关键词丰富、自包含）

## 2. 初始设计

### 2.1 整体工作流

设计了 6 步流水线，每步职责清晰：

```
第一步：解析文档
  docx → 文本 + 图片
  xlsx → markdown 表格
  md → 直接读取
  图片 → 逐张识别（多模态）

第二步：阅读理解
  完整阅读所有解析结果
  建立模块、流程、角色、规则的全景理解

第三步：填充功能需求文档
  对照标准模板逐节填充
  有内容就填，没有的标注 [待补充]

第四步：生成 FAQ 文档
  从需求中提炼常见问题
  按标准 FAQ 格式编写

第五步：识别缺失信息
  对照模板检查每个章节的完整性

第六步：生成待补充清单
  输出 JSON + Excel，方便产品经理逐项补充
```

### 2.2 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 解析工具 | Python 脚本 + Claude 多模态 | docx/xlsx 解析用 python-docx/openpyxl，图片识别利用 Claude 视觉能力 |
| 多版本处理 | 自动识别日期版本号，只保留最新 | 避免旧版本内容污染 |
| 图片处理策略 | 按图片类型区分：UI截图嵌入原图，流程图既嵌入也文字化 | UI截图文字化价值低，但流程图需要文字描述才能被 RAG 检索 |
| 缺失信息格式 | JSON → Excel 双输出 | JSON 便于程序处理，Excel 便于产品经理协作 |
| FAQ 设计 | 每条 FAQ 自包含（含常见问法、详细步骤、报错处理） | RAG 单个 chunk 能覆盖完整问答 |

### 2.3 Skill 文件结构

```
.claude/skills/standardize-wiki-docs/
├── SKILL.md              # 技能定义：6步工作流 + 填充规则 + 质量要求
├── parse_docs.py          # 文档解析脚本（docx/xlsx → 文本 + 图片）
└── generate_gaps.py       # 缺失清单生成脚本（JSON → Excel）
```

## 4. 标准模板设计

### 4.1 功能需求文档模板

设计了一套 RAG 友好的功能需求文档结构：

```
文档元数据（必填）     ← 关键词标签，便于检索
§1 需求概述            ← 背景、价值、目标用户
§2 功能总览            ← 功能列表 + 整体流程图
§3 功能详细说明         ← 按功能点展开（见后续迭代二优化）
§4 数据规则与权限       ← 字段定义表 + 权限矩阵
§5 界面原型            ← 页面位置 + 交互说明 + 原型链接
§6 非功能性需求         ← 性能、兼容性、安全
§7 依赖关系            ← 前置依赖 + 影响范围
§8 验收标准            ← 功能验收点 + 测试用例
§9 FAQ                ← 简要 Q&A（详细版在 FAQ 文档）
§10 附录              ← 相关文档 + 名词解释 + 变更记录
```

**RAG 优化设计**：
- 每个章节语义自包含，避免代词（"它"、"这个"）
- 元数据包含 3-5 个关键词，提升检索召回率
- 使用表格和列表等结构化格式，便于信息提取

### 4.2 FAQ 文档模板

每条 FAQ 自包含所有信息，保证 RAG 单个 chunk 能覆盖完整问答：

```markdown
### FAQ-ANALYSIS-001 如何查询分析任务？

**分类**：分析中心 > 查询功能
**关键词**：查询, 搜索, 筛选, 任务编号
**适用产品**：OmicsOne / v1.0

**常见问法**：          ← 多种表述方式，提升检索命中率
- 怎么查询分析任务？
- 如何搜索我的分析任务？
- 支持哪些查询条件？

**简要回答**：          ← 一段话直接回答
**详细步骤**：          ← 分步操作指南
**注意事项**：          ← 限制条件和注意点
**常见报错及处理**：     ← 错误信息 → 解决方案（单行，避免拆chunk）
**替代方案**：          ← 其他达到目的的方式
**相关问题**：          ← 关联 FAQ 引用
```

## 6. 迭代一：docx 图片提取

### 6.1 问题发现

首次应用后用户反馈：

> "这个 doc 文档里明明有一个'分析中心整体功能逻辑图'，转换成 md 格式时却没有了这部分内容"

### 6.2 根因分析

检查 `parse_docs.py` 源码，发现原始版本只提取了 `para.text`（纯文本）和表格，**完全忽略了 `w:drawing` XML 元素**（Word 中嵌入图片的容器）。

```python
# 原始代码：只提取文本
if tag == "p":
    text = para.text.strip()
    if text:
        parts.append(prefix + text)
    # ← 图片被跳过了！
```

用 Python 检查 docx 文件，确认有 3 张嵌入图片：
- img1：分析中心整体功能逻辑图（流程图，47KB）
- img2：重分析弹窗（UI截图，8KB）
- img3：停止分析确认弹窗（UI截图，5KB）

### 6.3 解决方案

为 `parse_docs.py` 新增图片提取功能：

```python
from docx.oxml.ns import qn

def _extract_images_from_paragraph(para_element, doc, images_dir, doc_stem, image_counter):
    """从段落元素中提取嵌入图片。"""
    refs = []
    # 查找 w:drawing 元素（Word 中嵌入图片的 XML 容器）
    drawings = para_element.findall('.//' + qn('w:drawing'))
    for drawing in drawings:
        # 查找 a:blip 元素（实际图片引用）
        blips = drawing.findall('.//' + qn('a:blip'))
        for blip in blips:
            embed_id = blip.get(qn('r:embed'))
            # 通过 relationship ID 获取图片二进制数据
            rel = doc.part.rels[embed_id]
            image_bytes = rel.target_part.blob
            # 保存图片并返回 markdown 引用
            img_path = images_dir / f"{doc_stem}_img{counter}.{ext}"
            img_path.write_bytes(image_bytes)
            refs.append(f"![{img_filename}](images/{img_filename})")
    return refs
```

**关键技术点**：
- `w:drawing` 是 Word 中嵌入对象（图片、图表等）的 XML 标签
- `a:blip` 是实际图片引用，通过 `r:embed` 属性关联到 relationship
- 通过 `doc.part.rels[embed_id]` 访问图片二进制数据
- 图片保存到 `_parsed/images/` 目录，文本中插入 markdown 图片引用

### 6.4 联动更新

图片提取能力加入后，需要更新多个文件：

| 文件 | 更新内容 |
|------|---------|
| `parse_docs.py` | 新增 `_extract_images_from_paragraph()` 函数 |

## 7. 迭代二：模板去重优化

### 7.1 问题发现

填写分析中心文档时发现**原模板的 §2 功能详细说明和 §3 功能规则存在大量重复描述**。

以"重分析"为例，需要在 3 个章节分别描述：

| 章节 | 内容 | 问题 |
|------|------|------|
| §2.2 功能流程 | 操作步骤描述 | 描述了操作流程 |
| §3.1 业务规则 | 触发条件 + 执行逻辑 | 又描述了一遍类似内容 |
| §5 异常处理 | 该功能的异常行 | 异常被混在公共表格里 |

**根因**：原模板按"信息类型"组织（流程归流程、规则归规则），而非按"功能点"组织。

### 7.2 头脑风暴过程

使用 `/brainstorming` 技能，通过 3 轮对话确定方案：

**第一轮：选择整体方案**

| 方案 | 思路 | 推荐 |
|------|------|------|
| A：加交叉引用 | 保持分离，加锚点链接 | 治标不治本 |
| **B：按功能点组织** | **每个功能点包含完整信息** | **推荐** |
| C：两层结构 | 简要版 + 详细版 | 容易写重复 |

→ 用户选择 **方案 B**

**第二轮：确认子章节结构**

| 选项 | 子章节 |
|------|--------|
| 精简版（4个） | 功能描述、操作流程、业务规则、异常处理 |
| 详细版（6个） | + 适用场景、输入输出 |
| 极简版（3个） | 合并更多内容 |

→ 用户选择**详细版但不要"输入输出"**，最终 **5 个子章节**

**第三轮：确认公共规则处理**

跨功能的数据规则（字段表）和权限规则（角色矩阵）不适合分散到各功能点 → 单独提取为 **§4 数据规则与权限**

### 7.3 新模板结构

```
§1 需求概述（不变）
§2 功能总览（新增：功能列表 + 整体流程图）
§3 功能详细说明（按功能点展开）
  3.X [功能点名称]
    3.X.1 功能描述
    3.X.2 适用场景
    3.X.3 操作流程（含 UI 截图）
    3.X.4 业务规则（仅该功能特有的）
    3.X.5 异常处理
§4 数据规则与权限（跨功能公共规则）
§5 界面原型（不变）
§6-§10（不变）
```

### 7.4 去重原则（写入模板编写指南）

| 内容类型 | 放在哪里 | 不放在哪里 |
|---------|---------|----------|
| 操作步骤、流程图、UI截图 | §3.X.3 操作流程 | 其他章节 |
| 该功能特有的业务逻辑 | §3.X.4 业务规则 | 其他功能点 |
| 该功能的异常场景 | §3.X.5 异常处理 | 公共异常表 |
| 字段定义、数据校验 | §4.1 数据规则 | 不分散到各功能点 |
| 角色权限矩阵 | §4.2 权限规则 | 不分散到各功能点 |

### 7.5 实施

同时更新 3 个文件：

1. **标准模板** `标准化模板-功能需求文档.md` — 重新设计章节结构
2. **Skill 定义** `SKILL.md` — 更新填充来源表
3. **已有文档** `功能需求文档.md` — 按新结构重写分析中心 6 个功能点

### 7.6 效果对比

| 指标 | 旧结构 | 新结构 |
|------|--------|--------|
| 理解一个功能需要翻阅的章节数 | 3 个 | 1 个 |
| RAG 能否用一个 chunk 覆盖完整功能 | 否 | 是 |
| 同一信息出现的次数 | 2-3 次 | 1 次 |

## 9. 经验总结

### 9.1 技能设计原则

1. **工具与判断分离**：机械的格式转换（docx→文本）交给 Python 脚本，需要理解和判断的工作（阅读理解、填充模板、识别缺失）交给 Claude
2. **双输出策略**：结构化数据（JSON）+ 人类友好格式（Excel/Markdown）并行输出，前者便于程序处理，后者便于协作
3. **图片按类型区分**：UI截图嵌入原图即可，流程图需要同时文字化（RAG 无法检索图片内容）
4. **模板自带编写指南**：去重原则、迁移指南写入模板末尾，让使用者（包括 Claude 自己）知道怎么正确填写

### 9.2 迭代改进模式

两次迭代都遵循了相同的模式：

```
在实际使用中发现问题
  → 定位根因（不是表面症状）
  → 设计解决方案
  → 实施变更（工具 + 模板 + 已有文档同步更新）
  → 验证效果
```

| 迭代 | 问题 | 根因 | 解决 |
|------|------|------|------|
| 一 | 流程图丢失 | parse_docs.py 忽略 w:drawing 元素 | 新增图片提取函数 |
| 二 | 文档重复描述 | 模板按信息类型而非功能点组织 | 重新设计模板结构 |

### 9.3 头脑风暴的关键

- **一次只问一个问题**：不要同时抛出多个选择
- **提供 2-3 个方案并推荐**：给用户决策权，但有明确推荐
- **分段确认**：每段 200-300 字确认一次，避免大返工
- **先结构后细节**：先定整体章节划分，再定子章节内容

### 9.4 可复用到其他场景

这个技能的架构可以复用到类似的文档标准化场景：

- 把标准模板替换为目标格式（如 API 文档模板、测试用例模板）
- 把 parse_docs.py 的解析能力扩展到其他格式（如 PDF、Confluence 导出）
- 把 generate_gaps.py 的缺失检测逻辑适配到新模板的章节结构
- FAQ 模板的自包含设计原则适用于任何 RAG 知识库
